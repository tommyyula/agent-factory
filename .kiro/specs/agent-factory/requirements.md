# Agent Factory Platform - 需求规格说明书

## 1. 项目概述

### 1.1 项目名称
Agent Factory Platform — 企业级 AI Agent 生产管理平台

### 1.2 项目描述
构建一个统一的单页 Web 应用，作为整个 Agent Factory 生态系统的门户和管理中心。该平台采用"就业/HR 隐喻"，管理 AI Agent 从"出生"到"就业"的完整生命周期。

### 1.3 业务背景
- **目标用户**: IT 管理者、领域专家、Agent 开发者、运维人员
- **应用场景**: 物流行业企业级 AI Agent 的统一生产、发布、部署和监控
- **价值主张**: 提供可视化、规范化的 Agent 管理平台，降低企业 AI 应用的复杂度

## 2. 用户故事 (EARS 格式)

### 2.1 Dashboard 模块

**US-001: 系统概览监控**
- **When** 用户登录平台时
- **If** 用户拥有查看权限
- **Then** 系统应该显示全局概览仪表盘，包括：
  - Agent 总数、活跃 Agent 数量、任务执行次数
  - 系统健康状态指标
  - 5层架构可视化（Data Sources → Ontology KB → Factory → Runtime → AaaS）
  - 实时活动流（最近 24 小时内的创建/部署/雇佣/异常活动）

**US-002: 快捷操作入口**
- **When** 用户在 Dashboard 页面时
- **If** 用户需要快速执行常用操作
- **Then** 系统应该提供快捷操作按钮：
  - 创建新 Agent
  - 查看 Ontology 状态
  - 访问 MarketPlace
  - 查看运行时监控

### 2.2 Ontology KB 模块

**US-003: Ontology 管理**
- **When** 领域专家需要管理知识图谱时
- **If** 用户拥有 Ontology 编辑权限
- **Then** 系统应该支持：
  - 查看 6 个物流行业知识域（WMS/TMS/FMS/HRM/YMS/OMS）
  - 显示知识采集状态（Ingestion/Modeling/Labeling/Reflection）
  - 提供概念/关系/事件链的可视化浏览（React Flow）
  - 展示 OntologyBlueprint.json 版本历史

**US-004: 知识统计分析**
- **When** 管理者需要评估知识图谱质量时
- **If** 系统包含有效的 Ontology 数据
- **Then** 系统应该显示：
  - 概念数量、关系数量统计
  - CQ（Competency Questions）覆盖率
  - 知识完整性评分

### 2.3 IDE 模块

**US-005: Agent 创建向导**
- **When** 开发者需要创建新 Agent 时
- **If** 用户拥有 Agent 创建权限
- **Then** 系统应该提供：
  - 从 Ontology 子集选择相关知识域
  - 基于需求描述自动生成 Agent 定义
  - Skills 配置界面
  - Prompt 编辑器

**US-006: Agent 开发环境**
- **When** 开发者编辑 Agent 配置时
- **If** Agent 处于开发状态
- **Then** 系统应该支持：
  - SDD 文件查看（requirements/design/domain-analysis/tasks）
  - 实时构建/编译状态追踪
  - Agent 测试和预览功能
  - 版本控制和回滚

### 2.4 MarketPlace 模块

**US-007: Agent 浏览和发现**
- **When** 用户需要寻找合适的 Agent 时
- **If** MarketPlace 包含已发布的 Agent
- **Then** 系统应该提供：
  - 卡片式 Agent 展示
  - 多维度搜索和过滤（标签/分类/评分）
  - 分类导航（按行业：WMS/TMS/HRM；按功能：数据分析/流程自动化/客服）

**US-008: Agent 雇佣流程**
- **When** 用户决定使用某个 Agent 时
- **If** 用户拥有足够的权限和预算
- **Then** 系统应该支持：
  - 查看 Agent 详细信息（描述、能力清单、评分、版本历史）
  - 配置 Agent 参数
  - Token 交换和支付流程
  - 一键部署到 Runtime

### 2.5 Runtime 模块

**US-009: 运行时监控**
- **When** 运维人员需要监控已部署的 Agent 时
- **If** 存在运行中的 Agent 实例
- **Then** 系统应该显示：
  - Agent 状态列表（运行/暂停/异常）
  - 资源使用情况和性能指标
  - 任务日志和对话历史
  - Job Agency 调度面板

**US-010: 安全治理**
- **When** 管理员需要管理平台安全时
- **If** 用户拥有管理员权限
- **Then** 系统应该提供：
  - Token 管理功能
  - RBAC 权限配置
  - 审计日志查看
  - Message Board 频道管理（#results/#coordination/#anomalies/#learnings）

## 3. 功能需求

### 3.1 核心功能
1. **Dashboard**: 全局监控和快捷操作
2. **Ontology KB**: 知识图谱管理和可视化
3. **IDE**: Agent 开发和测试环境
4. **MarketPlace**: Agent 发布和交易市场
5. **Runtime**: 运行时监控和管理

### 3.2 横切关注点
1. **身份认证和授权**: 基于角色的访问控制
2. **数据持久化**: IndexedDB 本地存储
3. **实时通信**: WebSocket 连接（模拟）
4. **国际化**: 中文界面支持
5. **主题管理**: 深色/浅色双模式

## 4. 非功能需求

### 4.1 性能需求
- **页面加载时间**: 首次加载 < 3秒，后续导航 < 1秒
- **响应时间**: 用户操作响应时间 < 500ms
- **数据加载**: 大型知识图谱渲染 < 5秒
- **内存使用**: 浏览器内存占用 < 500MB

### 4.2 可用性需求
- **浏览器兼容性**: Chrome 90+, Firefox 88+, Safari 14+
- **响应式设计**: 支持桌面端（1920x1080）和平板端（1024x768）
- **无障碍性**: 符合 WCAG 2.1 AA 标准
- **操作简便性**: 新用户在 15 分钟内完成基础操作培训

### 4.3 可靠性需求
- **数据一致性**: IndexedDB 数据同步机制
- **错误处理**: 优雅降级和错误恢复
- **离线支持**: 基础功能支持离线访问
- **数据备份**: 本地数据导入/导出功能

### 4.4 安全性需求
- **数据隔离**: 用户数据严格隔离
- **输入验证**: 所有用户输入进行验证和清理
- **XSS 防护**: 防止跨站脚本攻击
- **敏感信息**: 不在客户端存储敏感密钥

### 4.5 扩展性需求
- **模块化架构**: 支持功能模块独立开发和部署
- **API 抽象**: 预留后端 API 集成接口
- **插件化**: 支持第三方扩展和集成
- **国际化**: 支持多语言切换框架

## 6. 约束和限制

### 6.1 技术约束
- 必须使用 React 19 + TypeScript + Vite
- 必须使用 Tailwind CSS v4 + shadcn/ui
- 必须使用 IndexedDB 作为唯一数据存储
- 必须支持 oklch 色彩系统
- 必须使用 Zustand 进行状态管理

### 6.2 数据约束
- 全部使用 Mock 数据，不得连接真实后端
- Mock 数据必须足够丰富以支持完整 Demo
- 数据结构必须符合物流行业真实场景
- 预置数据包含完整的演示流程

### 6.3 UI/UX 约束
- 默认深色主题
- 必须支持中文界面
- 左侧导航栏布局
- 符合 ITEM 统一前端规范
- 参考 ifc-cli 脚手架模式

### 6.4 性能约束
- 单页应用总体积 < 5MB（gzipped）
- IndexedDB 数据库大小 < 50MB
- 知识图谱节点数量 < 5000
- 并发 Agent 监控数量 < 100

## 7. 关键风险和缓解策略

### 7.1 技术风险
**风险**: React Flow 大规模图谱渲染性能问题  
**缓解**: 实现虚拟化、分层加载和 LOD（细节层次）技术

**风险**: IndexedDB 跨浏览器兼容性问题  
**缓解**: 使用成熟的 Dexie.js 封装库，提供 fallback 机制

### 7.2 用户体验风险
**风险**: 复杂界面导致用户学习成本高  
**缓解**: 提供交互式导览、上下文帮助和渐进式信息披露

**风险**: 大量 Mock 数据影响真实感  
**缓解**: 基于真实物流场景设计数据，确保业务逻辑合理性

### 7.3 项目风险
**风险**: 功能范围过大导致交付延期  
**缓解**: 采用 MVP 优先策略，分阶段交付核心功能

**风险**: 缺乏真实用户反馈  
**缓解**: 内部用户测试和原型验证，收集早期反馈
